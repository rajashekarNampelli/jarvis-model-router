Callback / Hook System

Lifecycle Callbacks

NEW code_puppy pattern

13 lifecycle phases that plugins can hook into, enabling full extensibility without modifying core code.

  • startup / shutdown — app boot & teardown
  • pre_route / post_route — before/after model routing
  • pre_inference / post_inference — before/after LLM calls
  • pre_stream / post_stream — before/after streaming
  • provider_error — on provider failures
  • load_model_config — inject model configs
  • register_providers — register new LLM providers
  • register_health_checks — custom health checks
  • stream_event — real-time stream notifications

Blocking Callbacks

NEW Critical Feature

Callbacks can block operations by returning {"blocked": True, "reason": "..."}. The rate_limiter plugin uses this to throttle requests.

def _pre_inference_check(request, model_key): if not _limiter.is_allowed(model_key): return {"blocked": True, "reason": "Rate limit exceeded"} return None register_callback("pre_inference", _pre_inference_check)

Plugin Ownership Tracking

NEW code_puppy pattern

Every registered callback is tracked to its owning plugin. Disabled plugins have their callbacks automatically filtered out at fire time — no restart required.

  • set_loading_context(plugin_name) marks current plugin
  • get_callback_owner(func) returns plugin name
  • get_callbacks(phase) auto-filters disabled plugins

3-Tier Plugin System

Three Discovery Tiers

NEW code_puppy pattern

Plugins are auto-discovered from three tiers, loaded in order. Project plugins override user plugins on name collision.

  • Builtin — app/plugins/<name>/register_callbacks.py
  • User — ~/.jarvis/plugins/<name>/register_callbacks.py
  • Project — <CWD>/.jarvis/plugins/<name>/register_callbacks.py

Builtin Plugins Included

NEW

Two production-ready plugins ship with Jarvis:

  • rate_limiter — sliding window rate limiting per model (60 req/60s default). Blocks requests exceeding the limit via pre_inference callback.
  • request_logger — persists all inference/stream requests to ~/.jarvis/logs/requests.jsonl for auditing via post_inference / post_stream callbacks.

Plugin Management

NEW

Enable/disable plugins without code changes:

# Disable a plugin from app.plugins.config import disable_plugin disable_plugin("rate_limiter") # Re-enable it from app.plugins.config import enable_plugin enable_plugin("rate_limiter")

Disabled plugins are persisted to ~/.jarvis/disabled_plugins.json and filtered at callback fire time.

Event-Driven Hook Engine

Pattern-Based Event Matching

NEW code_puppy pattern

Declarative hook matching with compound expressions:

# Match any PreInference event "PreInference" # Match PreInference only for qwen model "PreInference && qwen" # Match either ProviderError or PostInference "ProviderError || PostInference" # Glob patterns "Pre*" "*Inference"

Command & Callback Hooks

NEW

Two hook types with different execution models:

  • command — runs a shell command via subprocess. Event data is passed as JSON on stdin. Exit code 0 = success, non-zero = block. Supports JSON block payloads on stdout.
  • callback — invokes a Python callable via module.path:function_name format. Async-aware with timeout support.

Blocking Verdicts

NEW Critical Feature

Hooks can block operations. Execution is sequential — if any hook blocks, subsequent hooks are skipped:

  • Command hooks: exit code != 0 blocks, stderr is the reason
  • Command hooks: stdout JSON {"decision": "block"} blocks
  • Callback hooks: returning {"blocked": True} blocks

Hook Configuration

NEW

Load hooks from a JSON config file via HOOKS_CONFIG_PATH env var:

{ "PreInference": [ { "matcher": "qwen", "hooks": [ { "type": "command", "command": "echo 'qwen called'", "timeout": 5000 } ] } ] }

Once Hooks & Hot Reload

NEW

once: true hooks fire only once per session. The engine tracks executed hook IDs. Config can be hot-reloaded via engine.reload_config() without restart.

Validation & Suggestions

NEW

Strict config validation with helpful error messages. Unknown event types, invalid matchers, and bad timeouts are caught at load time with actionable suggestions.

Resilience Patterns (MCP)

Circuit Breaker

NEW code_puppy pattern Critical

Prevents cascading failures when Ollama is down. Three states:

  • CLOSED — healthy, all requests pass through
  • OPEN — failing, all requests rejected immediately (no timeout wait)
  • HALF_OPEN — probing, allows one test request after cooldown

Default: 5 consecutive failures opens the circuit, 30s cooldown before probing, 2 successes to close.

Error Isolation

NEW code_puppy pattern

Categorizes errors and quarantines failing services:

  • TIMEOUT — provider didn't respond
  • CONNECTION — DNS/refused/network
  • HTTP_4XX — client errors (401, 403, 429 rate limit)
  • HTTP_5XX — server errors (500, 502, 503)
  • PARSE — response couldn't be decoded

After 5 errors in a category, the service is quarantined for 60s. Quarantined services fail-fast instead of waiting for timeouts.

Retry Manager

NEW code_puppy pattern

Exponential backoff with jitter to avoid thundering herd:

  • Default: 3 retries, 500ms base delay, 30s max delay
  • Per-service policy overrides
  • Jitter factor (10%) to desynchronize retries
  • Full observability via RetryStats
delay = min(base * 2^attempt, max_delay) delay += random(-jitter * delay, +jitter * delay)

Health Monitor

NEW code_puppy pattern

Periodic background health checks with rolling status:

  • HEALTHY — consecutive successes above threshold
  • DEGRADED — some failures but below threshold
  • UNHEALTHY — failures above threshold
  • UNKNOWN — no checks performed yet

Default: checks every 30s, 3 failures = unhealthy, 2 successes = healthy. Registers Ollama health check at startup.

429 Rate Limit Handling

NEW Critical Fix

Previously, HTTP 429 errors from Ollama were swallowed and re-thrown as generic 502 ProviderError. Now:

  • RateLimitError exception (HTTP 429) — proper status code
  • Categorized as HTTP_4XX in error isolation
  • Triggers circuit breaker failure recording
  • Fires provider_error callback for plugin notification
  • Rate limiter plugin provides client-side throttling

Key Advantages Summary

1. Zero-Code Extensibility

Add new behavior by dropping a single register_callbacks.py file. No core code changes needed. Plugins can be added at user level (personal) or project level (team-shared via git).

2. Production Resilience

When Ollama goes down, the circuit breaker opens in <1ms instead of every request waiting 120s to timeout. Error isolation quarantines failing services. Retry with backoff handles transient failures gracefully.

3. Proper 429 Handling

Rate limit errors (429) are now first-class citizens: proper HTTP 429 status code, categorized in error isolation, trigger circuit breaker recording, and fire provider_error callbacks. Client-side rate limiting plugin prevents hammering Ollama.

4. Agent Personality Switching

Different tasks need different routing strategies. QA Kitten always picks qwen for code analysis. Helios always picks deepseek for reasoning. JSON-defined agents let teams create custom personalities without Python.

5. Event-Driven Safety Net

The hook engine lets DevOps teams inject safety checks (blocking hooks) that run before every LLM call. Shell-command hooks mean even non-Python teams can add guards. Pattern matching means hooks can target specific models only.

6. Full Observability Stack

Circuit breaker stats, error categorization, quarantine status, retry statistics, health monitor history, hook execution results — all exposed via API and Prometheus. You know exactly what's failing and why.
